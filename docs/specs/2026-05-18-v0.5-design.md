# Notchy v0.5 — Workflow copilot

**Date:** 2026-05-18
**Status:** Draft → implementation
**Scope:** Tighter than v0.4. Three small features that target the user's actual
daily workflow (飞书 meetings, VSCode/Cursor coding, SSH to remotes).

---

## 1 · Tracks

### Track A — Meeting copilot

**Problem:** Every workday includes 3-5 meetings. The current Calendar widget
shows the next event but doesn't help when it's time to join. Users waste
20-40 seconds per meeting digging through the event detail for the join link.

**Solution:** Detect meeting URLs in event.location / event.notes; show a
countdown ("Meeting in 3 min") + a one-click **Join** button in the
dashboard when within 5 minutes of start.

**URL patterns to detect:**
- `lark.cn` / `feishu.cn` / `bytedance.com/meetings` (飞书)
- `zoom.us/j/`, `*.zoom.us`
- `meet.google.com`
- `teams.microsoft.com/l/meetup-join`
- `tencentmeeting.com`, `meeting.tencent.com`
- `webex.com`

**Behaviour:**
- Pre-meeting (≥5 min): existing UP NEXT row, no Join button
- Within 5 min: row shows "**Meeting in 3 min**" + "Join" button + accent colour shifts to amber
- During meeting (between start and end): "Live · ends in 27 min" + Join button stays
- 1 min before end: prompt fades

**No new permissions** — EventKit already granted.

### Track B — VSCode / Cursor project context

**Problem:** Notchy doesn't know what the user is working on. A dashboard
glance should tell you: "you're in Cursor, on `notchy/main` branch, last
edit 2 min ago." Useful when you `⌘-Tab` away for a meeting and come back.

**Solution:** Detect frontmost app (`NSWorkspace.frontmostApplication`). When
it's VSCode/Cursor/Xcode, parse the window title for the project name +
read the git branch from disk.

**Project name extraction:**
- **VSCode** title format: `<file> — <projectName>` or `<projectName> — Visual Studio Code`
- **Cursor** title format: `<file> - <projectName> — Cursor`
- **Xcode** title format: `<workspaceName> — <fileName>`

**Git branch:** walk up from the projectPath until `.git/HEAD` is found, read
the symbolic ref. Cache the (projectPath → branch) map for 30s.

**UI:** new dashboard row `💻 notchy · main` (project + git branch),
gated by `notchy.indicatorIDEContextEnabled` (default `true`).

### Track C — SSH session indicator

**Problem:** It's surprisingly common to forget you're in an SSH session
to a remote host and run a destructive command thinking you're local.
Boring engineer wins.

**Solution:** Periodically scan `ps -ax` for `ssh user@host` processes.
Show in dashboard which remote(s) the user has open + duration. Red badge
if hostname matches `prod` / `production` / `live` / `main` (heuristic).

**Detection:**
- Run `ps -axo pid,etime,command` every 30s
- Filter rows matching regex `^.*\b(?:ssh|mosh)\s+([^\s]+@[^\s]+|[^\s-]+\.[^\s]+)`
- Extract hostname from `user@host` or bare hostname

**UI:** new dashboard row `🔒 SSH · prod-server (2h 14m)` with red tint
when hostname matches a configurable danger pattern.

Setting `notchy.indicatorSSHEnabled` (default `true`).
Setting `notchy.indicatorSSHDangerPattern` (default `prod|production|live`).

## 2 · Non-goals

- **Lark unread message badge** — requires private API access to Lark's
  Dock badge or local SQLite database. Deferred to v0.6+ pending research.
- **AI task indicator** (Claude Code / Cursor working) — too fuzzy without
  hooks into the tools. Deferred.
- **Multi-window VSCode** — show only the frontmost project; if user has
  3 VSCode windows open, that's expected behaviour.

## 3 · Architecture

All three tracks fit the existing v0.4 `SystemStatusFeature` pattern:

```swift
@MainActor @Observable
final class SystemStatusFeature {
    // v0.4
    var chargingWatts: Int?
    var micInUse: PrivacyConsumer?
    // ... etc

    // v0.5
    var nextMeeting: Meeting?     // EventKit-derived w/ extracted URL
    var ideContext: IDEContext?   // frontmost VSCode/Cursor project + branch
    var sshSessions: [SSHSession] // active ssh user@host processes
}
```

Three new monitors:
- `MeetingURLExtractor` — pure function (no monitor needed; reuses Calendar bridge)
- `IDEContextMonitor` — 5s timer + frontmost-app notification listener
- `SSHMonitor` — 30s timer + ps -axo subprocess

Dashboard gains three new rows in the existing `extrasSection`, each gated
by its own Settings toggle.
