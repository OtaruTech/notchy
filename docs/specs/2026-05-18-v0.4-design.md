## 1 · Goals

Two parallel deliverables, both shipping in v0.4:

| Track | What | Why |
|-------|------|-----|
| **HUD takeover** | Replace macOS's centre-screen HUD for volume / brightness / keyboard backlight with a small slider hugging the notch | Signature notch-utility feature; closes biggest competitive gap vs NotchNook / AlcoveX |
| **System status indicators** (5×) | Charging wattage · Privacy dots (mic/cam) · Caffeine toggle · Network speed · BT multi-device battery | Five small high-frequency glanceables; every one of them is a 1× / day or more workflow win |

Every new feature gates behind a Settings toggle so users can pick what they want shown.

## 3 · Track 1 — HUD takeover

### 3.1 What gets replaced

| System event | Trigger | What we show |
|---|---|---|
| Volume up / down / mute | F11 / F12 / F10 / Touch Bar slider | Speaker icon + level bar |
| Brightness up / down | F1 / F2 (display) | Sun icon + level bar |
| Keyboard backlight up / down | F5 / F6 | Keyboard icon + level bar |

### 3.2 Why "takeover" needs scare-quotes

macOS displays its own HUD (the big rounded grey square mid-screen) via `OSDUIHelper.app`, a system process we cannot kill or hide cleanly. The textbook "disable system HUD" trick is `launchctl unload /System/Library/LaunchAgents/com.apple.OSDUIHelper.plist`, which requires SIP-disabled or special entitlements — **off the table for an ad-hoc-signed app**.

**Our approach: cover, don't suppress.** We show our own HUD inside the notch panel area the moment we detect the key press, with `level = .statusBar + 1` so it draws above the system OSD. The system OSD still fires, but it appears under our overlay and is barely visible behind the notch hardware. Practical result for the user: ours is the only HUD they see.

This is the same compromise NotchNook makes — they don't actually suppress the system HUD either.

### 3.3 Detecting the events

Two independent paths feed the same handler:

1. **CoreAudio / display brightness listeners** (preferred — fires reliably, no special entitlements):
   - Volume: `AudioObjectAddPropertyListenerBlock` on `kAudioHardwareServiceDeviceProperty_VirtualMainVolume` of the default output device
   - Brightness: `CGDisplayRegisterReconfigurationCallback` + `IOServiceGetMatchingService("AppleBacklightDisplay")` for the level value
   - Keyboard backlight: read via `IOServiceMatching("AppleHIDKeyboardEventDriverV2")` (private API but stable)

2. **Carbon hotkey monitor on F-row keys** (fallback / immediate trigger):
   - F10 / F11 / F12 / F1 / F2 / F5 / F6
   - When pressed, we show the HUD optimistically (using last-known level) and refresh after CoreAudio fires

### 3.4 UI

Reuses the existing `LiveActivityStrip` slot but with `HUDView` content. Single horizontal pill:

```
       ╭─ notch ─╮
  [🔊] ════════════════════  72%
   ↑   ↑ filled bar          ↑ percentage
```

- Width: 320pt, hugs bottom edge of notch
- Auto-dismisses 1.5s after the last key event
- Crossfade with album-art / waveform if music is also playing (HUD wins the slot for its 1.5s)

### 3.5 Settings

| Key | Default | Description |
|---|---|---|
| `notchy.hudVolumeEnabled` | `true` | Replace volume HUD |
| `notchy.hudBrightnessEnabled` | `true` | Replace brightness HUD |
| `notchy.hudKeyboardEnabled` | `true` | Replace keyboard-backlight HUD |
| `notchy.hudDuration` | `1.5` | Auto-dismiss seconds (slider 0.5–3.0) |

## 4 · Track 5 — System status indicators

Five small features. Each is independently shippable, each has its own Settings toggle. They appear primarily in the **Dashboard** view (replacing the current single CPU + battery row with a richer grid) and a subset in the **collapsed live-activity strip** when relevant.

### 4.1 ⚡ Charging wattage

**Source:** `IOPSCopyPowerSourcesInfo` → for each power source dictionary, read `kIOPSCurrentCapacityKey`, `kIOPSPowerSourceStateKey`, `kIOPSIsChargingKey`. Wattage: read `kIOPSAdapterDetailsKey` → `Watts` field (USB-C / MagSafe both expose this).

**UI:** Appears in `LiveActivityStrip` right wing when:
- AC plugged in **AND** `notchy.indicatorChargingEnabled` is on **AND** music isn't already using the wing

Pill: `⚡ 67W` (purple gradient, slow pulse animation while charging). Tooltip on hover: source type (MagSafe / USB-C / Smart Battery).

**Settings:** `notchy.indicatorChargingEnabled` (default `true`)

### 4.2 🔴 Privacy indicators

**Source:** macOS Sequoia 14.4+ has `NSWorkspace` notifications for mic/cam state. Lower-level fallback: `kAudioObjectPropertyMicrophoneActive` for mic; `CMIOObjectGetPropertyData` on cameras for camera.

For "which app", parse `tccutil` is not available — instead, on each fire, enumerate `NSWorkspace.shared.runningApplications` and find ones that requested mic/cam recently (via Audit Token / `kAudioObjectPropertyOwningProcess`).

**UI:** Two dots in the dashboard view (top right of clock area):
- 🟠 orange ⇒ microphone in use; hover: "X is using your microphone"
- 🟢 green ⇒ camera in use; hover: "Y is using your camera"

Optionally: a tiny dot in the collapsed live-activity strip's right wing when nothing else is there (i.e., user isn't playing music). Don't overlay album art.

**Settings:** `notchy.indicatorPrivacyEnabled` (default `true`)

### 4.3 ☕ Caffeine toggle

**Source:** spawn `caffeinate -d -i -m -u -s` (NSTask) on toggle-on; `kill -TERM` the process on toggle-off. `-d` prevents display sleep, `-i` system sleep, `-m` disk sleep, `-u` declares user activity for 5s, `-s` keeps system from sleeping on AC.

**UI:**
- Dashboard view: row with ☕ icon + "Keep awake" label + toggle switch
- Status bar: keyboard shortcut `⌘⌥K` toggles globally (Carbon hotkey)
- When active: small ☕ badge in collapsed strip's right wing (low priority — yields to charging / timer / waveform)

**Settings:** `notchy.indicatorCaffeineEnabled` (default `true` — but feature itself is off until user enables it)

### 4.4 📡 Network speed

**Source:** sample `/proc`-equivalent on Darwin: `getifaddrs` for all interfaces, sum `ifa_data.ifi_ibytes` / `ifi_obytes` across `en0`, `en1`, etc. Diff against previous sample (every 2s) → bytes/sec → human-readable KB/s or MB/s.

**UI:** Dashboard view: row with `↓ 5.2 MB/s   ↑ 1.1 MB/s`. Updates every 2s. Hidden when both rates < 50 KB/s and `notchy.indicatorNetworkHideIdle` is true (default true) to avoid noise.

**Settings:**
- `notchy.indicatorNetworkEnabled` (default `true`)
- `notchy.indicatorNetworkHideIdle` (default `true`)

### 4.5 🔋 BT multi-device battery

**Source:** `IOBluetoothDevice.pairedDevices()` filtered to connected devices. For each, `batteryPercentForMain` / `batteryPercentForLeft` / `batteryPercentForRight` / `batteryPercentForCase` where available (Apple devices report these; generic BT devices may only report a single battery level).

**UI:** Dashboard view: grid of icons + percentages:
- 🖱 Magic Mouse 47%
- ⌨️ Magic Keyboard 88%
- ⌚ Apple Watch 23%
- 🎧 AirPods L 60% / R 60% / Case 100%

(AirPods already get the burst notification on connect — this is the persistent view.)

**Settings:** `notchy.indicatorBTDevicesEnabled` (default `true`)

### 5.2 Dashboard redesign

The current dashboard has clock + date + next event + a single CPU/battery gauge row. The richer indicators need more room.

Proposed layout (panel 540×220):

```
┌─────────────────────────────────────────┐
│  09:47                          🟠 🟢   │   Clock + privacy dots
│  Monday, May 18                          │   Date
├─────────────────────────────────────────┤
│  📅 Standup in 13 min · Daily team sync │   Next event (existing)
├─────────────────────────────────────────┤
│  CPU 5%      Battery 67% ⚡67W           │   CPU + battery + wattage
│  ↓ 5.2 MB/s  ↑ 1.1 MB/s                  │   Network
│  🖱 47%   ⌨️ 88%   🎧 60/60/100%         │   BT devices
│  ☕ Keep awake [○]                       │   Caffeine toggle
└─────────────────────────────────────────┘
```

If user disables a row in Settings, the dashboard auto-collapses that row out of the layout.

### 5.3 New `SystemStatusFeature`

Single `@MainActor @Observable` orchestrator that owns:
- `PowerMonitor` (IOPS poll, 5s interval when on battery, 1s when charging)
- `PrivacyMonitor` (NSWorkspace notification subscriptions)
- `CaffeineController` (NSTask handle to current caffeinate process, if any)
- `NetworkMonitor` (getifaddrs poll, 2s interval)
- `BTBatteryMonitor` (IOBluetooth poll, 30s interval — battery doesn't change fast)

State exposed:
```swift
@Observable final class SystemStatusFeature {
    var chargingWatts: Int?              // nil = unplugged
    var micInUse: (appName: String)?     // nil = mic free
    var camInUse: (appName: String)?     // nil = cam free
    var isCaffeinated: Bool
    var networkUp: Double                // bytes/sec
    var networkDown: Double
    var btDevices: [BTDeviceBattery]     // [(name, level)]
}
```

### 5.4 New `HUDFeature` + `HUDView`

```swift
@MainActor @Observable
final class HUDFeature {
    var current: HUDEvent?  // nil = hidden

    struct HUDEvent { let kind: Kind; let level: Double }  // 0...1
    enum Kind { case volume, brightness, keyboardBacklight }
}
```

Drives a SwiftUI `HUDView` in the live-activity slot. Auto-dismiss timer set on every `current` update.

### 5.5 New state machine intent

`NotchIntent.hudEvent(HUDFeature.HUDEvent)` — does NOT cause a state expansion. The HUD draws in the collapsed strip area, no state transition needed.

## 6 · Risks

| # | Risk | Mitigation |
|---|------|-----------|
| R1 | OSDUIHelper still draws its HUD under ours | Position our overlay on top with `level = .statusBar + 1`. Document the small "double-render" if user moves Notchy off |
| R2 | Brightness IOService is private | Tested + stable since macOS 11. Fallback to F-key Carbon hotkey + last-known level if API access fails |
| R3 | IOPS adapter wattage field varies by adapter | Fall back to "Charging" text when watts unknown |
| R4 | Network bytes counter overflows on long-running interfaces | Use `UInt64` arithmetic; detect wraparound by `current < previous` |
| R5 | Privacy indicator "which app" is hard | Phase A: show the dot without app name. Phase B (post-v0.4): add app attribution. |
| R6 | Caffeinate process leaks if Notchy crashes | On launch, kill any orphaned `caffeinate -ds -i -m` we own (search by parent PID = old Notchy PID via launchd records). Acceptable trade-off: if crash + reboot, user just toggles again. |

## 9 · Future hooks (v0.5+ — not in this spec)

These hooks land naturally on top of v0.4:

- **HUD for AirPods connect** — extend `HUDFeature` with `case airpodsConnected(name)` already covers it
- **Lark unread badge** — add to dashboard row alongside BT devices, same orchestrator pattern as `SystemStatusFeature`
- **VSCode current project / git branch** — same observable feature pattern; tap `NSWorkspace.frontmostApplication` + `xattr`-stored project marker
- **AI task indicator** — `claude-code` / `cursor` long-running detection; live-activity strip slot

The architecture (one `@Observable` feature per data source, dashboard auto-composes rows) keeps adding features cheap.
